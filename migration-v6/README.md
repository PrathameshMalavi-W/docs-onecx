# Migration-V6: OneCX Angular 18 → 19

**Minimal agent design. Maximum autonomy. Full context preservation.**

A production-ready migration system that combines:
- ✅ **Minimum agents** (3 core agents only)
- ✅ **Full doc discovery** (visit EVERY link, no assumptions)
- ✅ **Skip functionality** (skip~N to mark N tasks complete)
- ✅ **Evidence collection** (8-field validation per task)
- ✅ **Robust error handling** (capture, map, resolve)
- ✅ **Real-world hardened** (10 actual failure modes fixed)

## 🗂️ Documentation Guide

### Why are there so many files in `docs/`?

This is **intentional**, not accidental.
Each file owns **one specific concern** so the agents do not have to read one huge mixed rulebook.

**Current split in this repo:**

- `AGENT-RULES.md` → shared behavior rules for all agents
- `HARD-RULES.md` → strict "must / must not" constraints
- `RUNTIME-DISCOVERY-PIPELINE.md` + `STRICT-DOC-EXPANSION.md` → planner/discovery behavior
- `EXECUTOR-PHASE-SUPPORT.md` + `NEVER-SKIP-ALWAYS-FIX-PROTOCOL.md` + `MULTI-PHASE-ERROR-TRACKING.md` → executor behavior and validation flow
- `VERSION-AWARE-UPGRADE-PROTOCOL.md` → Angular/Nx/PrimeNG upgrade rules
- `SKIP-FUNCTIONALITY.md` → orchestrator skip handling
- `CONTEXT-PRESERVATION-MANDATE.md` → progress/state discipline

**Why the files are large:**

- they include real migration failures,
- edge cases and anti-patterns,
- exact completion rules,
- and phase-specific validation behavior.

That size is there to stop the agents from guessing.

### Which agent uses which files?

**`migration-orchestrator-v6.agent.md`**
- Reads: `docs/AGENT-RULES.md`, `docs/HARD-RULES.md`, `docs/SKIP-FUNCTIONALITY.md`
- Purpose: command routing, phase gates, skip handling, when to ask the user

**`migration-planner-v6.agent.md`**
- Reads: `docs/AGENT-RULES.md`, `docs/HARD-RULES.md`, `docs/RUNTIME-DISCOVERY-PIPELINE.md`, `docs/STRICT-DOC-EXPANSION.md`
- Purpose: Phase 1 planning, link expansion, task discovery, no-assumption reading

**`migration-executor-v6.agent.md`**
- Reads: `docs/AGENT-RULES.md`, `docs/HARD-RULES.md`, `docs/EXECUTOR-PHASE-SUPPORT.md`, `docs/NEVER-SKIP-ALWAYS-FIX-PROTOCOL.md`, `docs/MULTI-PHASE-ERROR-TRACKING.md`, `docs/VERSION-AWARE-UPGRADE-PROTOCOL.md`
- Purpose: task execution, build/lint/test validation, post-migration recovery, version-aware upgrades

### If you need to add a new rule, where should it go?

| Rule type                        | Add it here first                                                                    | Why                                  |
| -------------------------------- | ------------------------------------------------------------------------------------ | ------------------------------------ |
| Global rule for every step       | `docs/AGENT-RULES.md` or `docs/HARD-RULES.md`                                        | applies to all agents and all tasks  |
| Planner / discovery rule         | `agents/migration-planner-v6.agent.md`                                               | owned by the planner                 |
| Executor task or validation rule | `agents/migration-executor-v6.agent.md`                                              | owned by the executor                |
| Post-migration / Phase C rule    | `docs/MULTI-PHASE-ERROR-TRACKING.md` and `agents/migration-executor-v6.agent.md`     | affects deferred validation behavior |
| Version / upgrade rule           | `docs/VERSION-AWARE-UPGRADE-PROTOCOL.md` and `agents/migration-executor-v6.agent.md` | affects upgrade logic                |
| Skip / fast-forward rule         | `docs/SKIP-FUNCTIONALITY.md` and `agents/migration-orchestrator-v6.agent.md`         | affects routing and skipping         |

> **Concrete rule:** if the rule is tied to one step, add it to the **owning agent file first**. If it is reusable or cross-cutting, also mirror it in the matching file under `docs/`.

### Concrete examples

#### Example A — rule for one specific step

If you want a rule like:

> "When a task migrates PrimeNG table filters, the executor must rerun build and the related tests before closing that task."

Add it first to:

- `agents/migration-executor-v6.agent.md`

Then optionally mirror/explain it in:

- `docs/EXECUTOR-PHASE-SUPPORT.md`
- `docs/MULTI-PHASE-ERROR-TRACKING.md`

Example snippet:

```markdown
### PrimeNG table migration rule
- If the current task changes `p-table` filtering, rerun build and the related tests before marking the task complete.
- In Phase C, if the failure is clearly deferred to a later cleanup step, record it and revisit it in final validation.
```

#### Example B — generic rule for every step

If you want a rule like:

> "Every migration task must record the exact validation command and output before it can be marked complete."

Add it first to:

- `docs/HARD-RULES.md` or `docs/AGENT-RULES.md`

Then mirror it in agent files if execution behavior changes.

Example snippet:

```markdown
### H21: Validation Evidence Must Be Explicit
- ✅ Record the exact validation command that was run
- ✅ Record whether `build`, `lint`, and `test` passed or failed
- ❌ Do NOT claim success without fresh output evidence
```

## 🔄 Workflow Overview

```
Phase 1: Preparation & Planning
├─ Audits: npm install, npm test, configuration
├─ Discovery: Fetch ALL OneCX/PrimeNG/Nx docs (visit every link)
├─ Task creation: Count subsections, create one task per
└─ Result: MIGRATION_PROGRESS.md with full task tree

Phase A: Pre-Migration Execution
├─ Task 1: npm run build, collect evidence, validate
├─ Task 2: Task 3: ...
└─ Result: All Phase A tasks marked [x]

Phase B: Manual Developer Confirmation
├─ Developer runs: npm build, npm lint, npm test locally
├─ Developer confirms: All tests pass
└─ Developer signs off in MIGRATION_PROGRESS.md

Phase C: Post-Migration Execution  
├─ Task 1: npm run build, validate
├─ Task 2: Task 3: ...
└─ Result: Migration complete, tests pass
```

## ✨ Key Features

### 1. Minimal Agent Count
- **3 agents only** (orchestrator, planner, executor)
- No special agents per task type
- Executor handles ALL Phase A & C tasks
- Delegation pattern: Router → Specialist → Executor

### 2. Strict Documentation Expansion
Planner enforces:
- ✅ Visit EVERY link on index page
- ✅ Fetch FULL page content (not summaries)
- ✅ Count all H2 subsections
- ✅ Fetch sub-links if present
- ❌ NO assumptions from headlines

### 3. Skip~N Functionality
```
Skip~3  →  Mark tasks 1,2,3 as "[-] not applicable"  →  Jump to task 4
```
Use when:
- Task is conditional and doesn't apply
- Already completed manually before migration started
- Task is optional (nice-to-have)

### 4. Evidence Collection
Each task records:
1. Source pages (URLs visited)
2. Applicability decision (must-have / nice-to-have / not applicable)
3. Repository evidence (grep results, inspection)
4. Planned action (exactly what was done)
5. Files changed (exact list)
6. Validation (npm build, npm lint, npm test output)
7. Final outcome (success / blocked / error)
8. Edge cases (gotchas found)

### 5. Error Handling
- Capture full error output
- Map to root cause
- Never hide failures
- Leave task [ ] (needs rework)
- Next run: retry with fix

## 🎯 Design Principles

### Minimal
- 3 agents max (no N-agent sprawl per task type)
- Each agent has single, clear responsibility
- Agents delegate, don't duplicate

### Delegation
- Orchestrator routes commands
- Planner discovers once
- Executor autonomously executes
- No agent calls another agent (only orchestrator delegates)

### Consistent
- Single state file: MIGRATION_PROGRESS.md
- 3 markers only: [ ] not started, [x] completed, [-] not applicable
- 8-field evidence required before [x]

### Honest
- Full error output captured
- No hidden failures
- Gap in task tree? Flag it
- Task failed? Leave [ ] and explain
- Unknown applicability? Ask permission

### Autonomous
- Executor runs per-task, no hand-holds
- Validate at task level, not step level
- Evidence collected automatically
- Minimal human intervention needed
